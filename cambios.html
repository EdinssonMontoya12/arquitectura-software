1. REDIRECCIONAMIENTO DE NUBE POR SOBRECARGA

En el momento que una nube deje de funcionar de la manera adecuada o no pueda soportar la cantidad de peticiones realizadas por lo usuarios, la carga debe ser redirigida a la segunda nube pública que cuenta con la aplicación ya desplegada y réplicas de la base de datos.

Definir un servidor principal con el software de Nginx por el cual pasan todas las peticiones y de esta forma detectar cuando la nube principal está saturada o sufriendo fallas para redirigir todas las peticiones hacia la nube secundaria automáticamente, sin intervención humana y realizando el proceso de forma que el usuario no note la redirección.

  • Al centralizar el llamado de peticiones a un único servidor, si este llegase a fallar por cuestiones ajenas al equipo de desarrollo (el proveedor de nube está sufriendo fallas), el servicio quedaría inactivo de forma indeterminada hasta montar un nuevo servicio de redirección y redireccionar los DNS a la nueva IP.

2. USO DE CONTENEDORES PARA EL DESPLIEGUE DE APLICACIONES

El código de la aplicación es ejecutado directamente en la nube con el servidor configurado para ejecutar ese código en específico.

Cambiar el sistema de despliegue y ejecución en servidor de las aplicaciones, se recomienda usar contenedores basados en Docker, de esta forma podemos desplegar el código en diferentes nubes independiente de su configuración, siempre que logre ejecutar Docker el servicio se ejecutará de forma correcta sin necesidad de modificar el código fuente.

  • Se debe tener en cuenta qué tecnologías se usan dentro del código fuente debido a que algunas son compatibles únicamente con el entorno de Windows y los contenedores por defecto ejecutan internamente un ambiente Linux.
  • Se hace necesario usar un repositorio de imágenes (Docker Hub, Kubernet, GitHub Actions).

3. SEPARACIÓN DE SERVIDORES CÓDIGO Y BASE DE DATOS

La nube almacena y ejecuta internamente tanto el servidor del código como el servidor de la base de datos.

Creación de un nuevo servidor para realizar la separación del código y la base de datos de esta forma se distribuye la carga entre dos nubes distintas para dos procesos distintos con distintos procesos, se elimina la dependencia que tiene el código de alojarse en la misma nube que la base de datos, además de que se pueden crear nubes especializadas para la ejecución de los diferentes servicios aumentando así su rendimiento y reduciendo el riesgo de caídas.

  • Al tener una nueva nube se debe contar con personal más capacitado para administrarlas de forma correcta.
  • Si tanto la nube de base de datos principal como la réplica fallan el servicio no estaría disponible hasta levantar una instancia de la base de datos.

4. RÉPLICAS DE LA BASE DE DATOS

Se requiere que la información sea precisa y esté en sincronía entre la nube principal y la de respaldo.

Se crea un nuevo servidor de base de datos que estará esperando el momento de ser usado, esta nueva instancia se estará sincronizado en tiempo real con el servidor principal con el fin de ofrecer al los usuario información actualizada.

  • La latencia de actualización de datos de la base de datos réplica puede aumentar la latencia hasta 3 min dependiendo de la cantidad de información y la complejidad de los procesos.
  • Algunas versiones de MySql tienen bugs que pueden inducir el lag dentro de la base de datos principal, la versión 8.0.33 aumenta de forma gradual el consumo de ram de la base de datos principal cuando la réplica está habilitada.

5. AMAZON S3 COMO ALMACENAMIENTO EN BLOQUE

Se hace necesario que el sistema de almacenamiento por bloques esté sincronizado entre ambas nubes.

Se consume el servicio Amazon S3 para el sistema de almacenamiento por bloques, esto protegerá el almacenamiento de datos evitando su pérdida sin previa autorización o por fallas humanas y permitirá gran cantidad de peticiones ya sean para cargar o descargar ficheros sin colapsar.

  • El volumen de datos aumentará de forma gradual en el tiempo lo que obligará a mejorar o reorganizar el sistema como se está estructurando el almacén de ficheros.
